v3.0.0: Explicit entity configuration with safety controls

BREAKING CHANGES:
- Requires explicit configuration in scheiber.yaml to expose entities
- MQTT Discovery pattern changed to standard HA format
- Unique ID suffix changed from _v2 to _v3

This major version introduces explicit configuration for MQTT Discovery,
addressing safety concerns about accidental control of critical systems.

What's New:
============

1. **Explicit Entity Configuration (scheiber.yaml)**
   - New YAML configuration file required in /config/scheiber.yaml
   - Defines which Bloc9 outputs to expose via MQTT Discovery
   - Per-entity control: name, entity_id, classification (light/switch)
   - Safety: Critical systems can be omitted from discovery
   - Example configuration provided in README.md

2. **Standard Home Assistant Discovery Pattern**
   - Old: homeassistant/scheiber/bloc9/{bus_id}/{output}/config
   - New: homeassistant/{component}/{object_id}/config
   - Follows HA convention for discovery topics
   - State/command topics remain under homeassistant/scheiber/* namespace

3. **Config Validation & Graceful Degradation**
   - New config_loader.py module with comprehensive validation
   - Validates bus_id, output format (s1-s6), required fields
   - Helpful error messages for configuration issues
   - Without config: bridge works but no HA discovery

Architecture Changes:
=====================

New Files:
- scheiber.yaml: Entity configuration (240 lines, 10 Bloc9 devices)
- config_loader.py: Configuration loader with validation (220 lines)

Modified Files:
- devices.py: Accept discovery_configs parameter, rewrite discovery
- mqtt_bridge.py: Load config, pass to devices, --config-file argument
- run.sh: Pass --config-file /config/scheiber.yaml
- config.yaml: Map /config directory, bump to v3.0.0
- README.md: Comprehensive configuration documentation

Key Classes:
- DiscoveryConfig: Single entity configuration
- ScheiberConfig: Container with get_bloc9_configs(bus_id) method
- load_config(path): Returns ScheiberConfig or None

Data Flow:
scheiber.yaml → config_loader → mqtt_bridge → devices → discovery

Migration from v2.x:
====================

Old Behavior (v2.x):
- All 6 outputs on every Bloc9 automatically exposed as lights
- Custom discovery pattern: homeassistant/scheiber/bloc9/{id}/{output}/config
- No control over entity exposure

New Behavior (v3.0.0):
- Only configured outputs exposed
- Standard HA pattern: homeassistant/{component}/{object_id}/config
- Explicit control: lights vs switches, custom names
- Safety: Critical systems can be excluded

Migration Steps:
1. Create /config/scheiber.yaml in Home Assistant config directory
2. Define Bloc9 devices with bus_id and name
3. List outputs to expose under lights: or switches:
4. Specify name, entity_id, and output (s1-s6) for each entity
5. Restart addon

Example scheiber.yaml:
```yaml
bloc9:
  - bus_id: 7
    name: "Salon Bloc9"
    lights:
      - name: "Working Light"
        entity_id: "light.salon_working_light"
        output: s1
    switches:
      - name: "Water Pump"
        entity_id: "switch.salon_water_pump"
        output: s3
```

Without scheiber.yaml:
- Bridge still monitors CAN and publishes to MQTT
- No entities appear in Home Assistant (no discovery)
- Warning logged at startup

Safety Rationale:
=================

This change addresses the risk of accidentally triggering critical
systems (bilge pumps, navigation lights, refrigeration) while
navigating Home Assistant UI or experimenting with automations.

By requiring explicit configuration:
- Users consciously decide what to expose
- Critical systems can be safely omitted
- Entity names are descriptive (prevents confusion)
- Component type (light/switch) matches actual load
